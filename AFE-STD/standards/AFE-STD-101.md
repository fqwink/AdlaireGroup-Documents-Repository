## 1. 概要・適用範囲

### 1.1 概要

AFE-STD-101 は、Adlaire Framework Ecosystem（AFE）およびその関連プロジェクトにおける
**オートローディング規則**を定義する標準仕様書である。

本標準は PHP-FIG PSR-4 への依存を完全に排し、AFE の設計思想
（モジュラーコンポーネント駆動・APIインターフェース駆動・DIコンテナ駆動）に最適化された
ファイルマッピング・ディレクトリ構造・オートローダー実装規則を規定する。

AFE-STD-101 は **AFE-STD-100（命名規則）と密接に連携**する。
AFE-STD-100 で定めたクラス命名規則・名前空間規則を物理ファイルシステムへ正確に投影するための
規則を本標準が提供する。

### 1.2 適用範囲

本標準は以下のすべてに適用される。

- Adlaire Framework Ecosystem（AFE）のソースコード
- AFE を基盤として開発されるすべてのプロジェクト
- Adlaire Group の内部システム

### 1.3 準拠レベル定義

| キーワード | 意味 |
|-----------|------|
| **必須（MUST）** | 例外なく遵守すること ✅ |
| **禁止（MUST NOT）** | 例外なく行わないこと ❌ |
| **推奨（SHOULD）** | 特別な理由がない限り遵守すること |
| **非推奨（SHOULD NOT）** | 特別な理由がない限り行わないこと |

## 2. PSR-4との差別化ポイント（総括）

PSR-4 と AFE-STD-101 の主要な差別化ポイントを示す。

| 項目 | PSR-4（参考） | AFE-STD-101 | 差別化レベル |
|------|-------------|------------|------------|
| **ルートディレクトリ名** | 任意（推奨なし） | `src/` **必須**（変更禁止） | 🔴 AFE独自 |
| **サブディレクトリ命名** | 任意（任意の文字列） | **StudlyCaps 必須**（AFE-STD-100 と連動） | 🔴 AFE独自 |
| **レイヤー別ディレクトリ構造** | 未規定 | **7レイヤー固定構造を必須**定義 | 🔴 AFE独自 |
| **インターフェース配置** | 未規定 | 実装クラスと**同一ディレクトリ必須** | 🔴 AFE独自 |
| **抽象クラス配置** | 未規定 | 実装クラスと**同一ディレクトリ必須** | 🔴 AFE独自 |
| **Contracts ディレクトリ** | 未規定（慣習のみ） | **禁止**（AFE独自配置ルール） | 🔴 AFE独自 |
| **Abstracts / Base ディレクトリ** | 未規定（慣習のみ） | **禁止**（AFE独自配置ルール） | 🔴 AFE独自 |
| **複数ルート名前空間マッピング** | 許可（任意の数） | **最大3プレフィックスまで**制限 | 🔴 AFE独自 |
| **ファイル拡張子** | `.php` 推奨 | `.php` **必須**（他拡張子禁止） | 🟡 強化 |
| **クラスファイルの1定義制限** | 推奨 | **必須**（AFE-STD-100 Section 3.3 参照） | 🟡 強化 |
| **名前空間とディレクトリの完全一致** | 必須 | 必須（同一） | 🟢 同一 |
| **大文字小文字の完全一致** | 必須（大文字小文字区別） | 必須（同一） | 🟢 同一 |
| **Composerオートローダー利用** | 任意（標準的利用） | **必須**（独自オートローダー実装禁止） | 🔴 AFE独自 |

## 3. ディレクトリ構造規則

### 3.1 ルートディレクトリ（**PSR-4との主要差別化点**）

PSR-4 はオートローディングのルートディレクトリ名を任意としているが、
AFE-STD-101 は `src/` を唯一のルートソースディレクトリとして**必須**とする。

| 規則 | 準拠レベル | 内容 |
|------|----------|------|
| プロダクションコードのルートは `src/` に統一する | **必須** | `lib/`, `app/`, `source/` 等は禁止 |
| テストコードのルートは `tests/` に統一する | **必須** | `test/`, `spec/` 等は禁止 |
| `src/` と `tests/` は**プロジェクトルート直下**に配置する | **必須** | ネストした位置への配置は禁止 |

```
✅ 正しいプロジェクト構造
project-root/
├── src/           ← プロダクションコード（必須・変更禁止）
├── tests/         ← テストコード（必須・変更禁止）
├── composer.json
├── .gitignore
└── README.md

❌ 誤り
project-root/
├── lib/           ← 禁止（src/ を使用する）
├── app/           ← 禁止（src/ を使用する）
├── source/        ← 禁止（src/ を使用する）
└── test/          ← 禁止（tests/ を使用する）
```

### 3.2 サブディレクトリ命名（**PSR-4との主要差別化点**）

PSR-4 はサブディレクトリ名の命名形式を規定しないが、
AFE-STD-101 は AFE-STD-100 の命名規則と連動させ **StudlyCaps を必須**とする。

| 規則 | 準拠レベル | 内容 |
|------|----------|------|
| すべてのサブディレクトリ名は **StudlyCaps** で命名する | **必須** | 名前空間セグメントと完全一致させる |
| スネークケース（`http_message/`）のディレクトリ名 | **禁止** | — |
| ケバブケース（`http-message/`）のディレクトリ名 | **禁止** | — |
| 全小文字（`httpmessage/`）のディレクトリ名 | **禁止** | — |

```
✅ 正しいディレクトリ命名
src/
  Http/
    Message/
    Middleware/
  Database/
    Orm/
  Auth/
    Rbac/
  Container/
  Event/
  Logger/

❌ 誤り
src/
  http/            → 全小文字
  http_message/    → スネークケース
  http-message/    → ケバブケース
  HttpMessage/     → 2単語の誤り結合は許容（StudlyCapsなので正しい形は HttpMessage）
```

## 4. 名前空間マッピング規則

### 4.1 名前空間とディレクトリの完全一致

PSR-4 と同様に、名前空間とディレクトリ構造を完全に一致させる。
AFE-STD-101 はこれを**強制**とし、例外を一切認めない。

| 規則 | 準拠レベル |
|------|----------|
| 名前空間のセグメントとディレクトリ名を**完全一致**（大文字小文字含む）させる | **必須** |
| 名前空間に存在しないディレクトリを作成する | **禁止** |
| ディレクトリに対応しない名前空間セグメントを定義する | **禁止** |

### 4.2 AFE コアの名前空間マッピング定義

AFE フレームワークのコアは以下のマッピングを基本とする。

| 名前空間プレフィックス | ベースディレクトリ |
|---------------------|----------------|
| `Adlaire\Framework\` | `src/` |

```
名前空間                                → ファイルパス
Adlaire\Framework\Http\Message\Request  → src/Http/Message/Request.php
Adlaire\Framework\Container\Container   → src/Container/Container.php
Adlaire\Framework\Logger\Logger         → src/Logger/Logger.php
Adlaire\Framework\Event\EventDispatcher → src/Event/EventDispatcher.php
Adlaire\Framework\Auth\Authenticator    → src/Auth/Authenticator.php
```

### 4.3 名前空間ルートの定義

| 規則 | 準拠レベル | 内容 |
|------|----------|------|
| AFE フレームワークのルート名前空間は `Adlaire\Framework\` を使用する | **必須** | — |
| AFE を使用するアプリケーションのルート名前空間は `{組織名}\{プロジェクト名}\` とする | **必須** | 例: `AdlaireGroup\AdlaireCms\` |
| ルート名前空間は StudlyCaps で命名する | **必須** | AFE-STD-100 Section 5.2 参照 |

## 5. オートローダー実装規則

### 5.1 Composerオートローダー採用（**PSR-4との主要差別化点**）

AFE-STD-101 は Composer のオートローダーを**唯一の実装手段**として採用する。
PSR-4 は「準拠するオートローダーの実装方法」を定めるにとどまるが、
AFE-STD-101 は実装手段を Composer に**完全固定**する。

| 規則 | 準拠レベル | 内容 |
|------|----------|------|
| Composer の `autoload` 設定を使用してオートローディングを構成する | **必須** | — |
| 独自のオートローダークラス実装（`spl_autoload_register` 等） | **禁止** | Composer で代替する |
| `require` / `include` による手動ファイル読み込み（クラスロード目的） | **禁止** | Composer で代替する |
| `composer dump-autoload` を CI/CD パイプラインに組み込む | **必須** | — |

### 5.2 オートロードキャッシュ

| 規則 | 準拠レベル | 内容 |
|------|----------|------|
| プロダクション環境では `composer dump-autoload --optimize` を使用する | **必須** | クラスマップによるキャッシュを有効化 |
| 開発環境では `composer dump-autoload` （最適化なし）を使用する | **推奨** | ファイル変更の即時反映を優先する |

```bash
# ✅ プロダクション環境
composer dump-autoload --optimize

# ✅ 開発環境
composer dump-autoload

# ❌ 禁止: spl_autoload_register による手動実装
spl_autoload_register(function (string $class): void {
    // 手動オートローダー実装は禁止
});
```

## 6. Composerオートローディング設定規則

### 6.1 `composer.json` の `autoload` セクション記述規則

| 規則 | 準拠レベル | 内容 |
|------|----------|------|
| `autoload.psr-4` キーを使用してオートローディングを設定する | **必須** | — |
| 名前空間プレフィックスの末尾には `\\` を付与する | **必須** | JSON 内では `\\\\` とエスケープする |
| ベースディレクトリの末尾には `/` を付与する | **必須** | — |
| `autoload.classmap` の使用 | **禁止** | 名前空間マッピングのみ許可 |
| `autoload.files` の使用（クラス定義ファイル） | **禁止** | クラスは必ず名前空間マッピングで解決する |
| `autoload.files` の使用（ヘルパー関数ファイル） | **推奨** | 関数定義ファイルに限り許可 |

```json
{
    "autoload": {
        "psr-4": {
            "Adlaire\\Framework\\": "src/"
        },
        "files": [
            "src/helpers.php"
        ]
    },
    "autoload-dev": {
        "psr-4": {
            "Adlaire\\Framework\\Tests\\": "tests/"
        }
    }
}
```

> ⚠️ `autoload.psr-4` キー名は Composer の仕様上 `psr-4` を使用するが、
> これは AFE-STD-101 の「PSR-4 採用」を意味しない。
> Composer の設定インターフェースとして `psr-4` キーを利用するのみであり、
> オートローディングの**実質的な規則は AFE-STD-101 が定義**する。

### 6.2 名前空間プレフィックスの定義規則

| 規則 | 準拠レベル | 内容 |
|------|----------|------|
| `autoload.psr-4` に定義する名前空間プレフィックスは**最大3つまで**とする | **必須** | Section 8 参照 |
| プレフィックスは具体的なベースディレクトリに1対1でマッピングする | **必須** | 1プレフィックスが複数ディレクトリにマップすることは禁止 |

## 7. レイヤー別ディレクトリ規則（AFE独自）

### 7.1 AFEコアのレイヤー構造（**PSR-4との主要差別化点**）

PSR-4 はディレクトリ構造の意味・レイヤー設計を一切規定しないが、
AFE-STD-101 は AFE コアが採用すべきレイヤー別ディレクトリ構造を**必須**として定義する。

```
src/
├── Http/                  ← HTTPレイヤー（AFE-STD-200〜202 準拠）
│   ├── Message/           ← リクエスト・レスポンス
│   ├── Middleware/        ← ミドルウェア群
│   └── Factory/           ← HTTPオブジェクトファクトリー
│
├── Container/             ← DIコンテナ（AFE-STD-300 準拠）
│
├── Logger/                ← ロガー（AFE-STD-400 準拠）
│
├── Event/                 ← イベントシステム（AFE-STD-401 準拠）
│
├── Cache/                 ← キャッシュ（AFE-STD-500〜501 準拠）
│
├── Database/              ← データベース・ORM（AFE-STD-600 準拠）
│   └── Orm/
│
├── Auth/                  ← 認証・認可（AFE-STD-601 準拠）
│   └── Rbac/
│
├── Validation/            ← バリデーション（AFE-STD-602 準拠）
│
├── Console/               ← CLIツール（AFE-STD-603 準拠）
│
├── Exception/             ← フレームワーク共通例外クラス群
│
└── Support/               ← ユーティリティ・ヘルパー群
```

### 7.2 インターフェース・抽象クラスの配置規則（**PSR-4との主要差別化点**）

PSR-4 はインターフェースや抽象クラスの配置場所を規定しないため、
多くのフレームワーク・プロジェクトでは `Contracts/`・`Abstracts/`・`Interfaces/` などの
**専用サブディレクトリ**を作る慣習がある。

AFE-STD-101 はこれを**明示的に禁止**し、実装クラスと同一ディレクトリへの配置を**必須**とする。

| 規則 | 準拠レベル | 内容 |
|------|----------|------|
| インターフェースは**実装クラスと同一ディレクトリ**に配置する | **必須** | — |
| 抽象クラスは**実装クラスと同一ディレクトリ**に配置する | **必須** | — |
| `Contracts/` ディレクトリの作成 | **禁止** | インターフェースの隔離配置は禁止 |
| `Interfaces/` ディレクトリの作成 | **禁止** | インターフェースの隔離配置は禁止 |
| `Abstracts/` ディレクトリの作成 | **禁止** | 抽象クラスの隔離配置は禁止 |
| `Base/` ディレクトリの作成 | **禁止** | 抽象クラスの隔離配置は禁止 |

```
✅ 正しい: インターフェースと実装を同一ディレクトリに配置
src/
  Container/
    ContainerInterface.php      ← インターフェース
    AbstractContainer.php       ← 抽象クラス
    Container.php               ← 実装クラス

src/
  Logger/
    LoggerInterface.php         ← インターフェース
    AbstractLogger.php          ← 抽象クラス
    Logger.php                  ← 実装クラス

❌ 禁止: 専用サブディレクトリへの隔離配置
src/
  Container/
    Contracts/
      ContainerInterface.php    ← Contracts/ ディレクトリは禁止
    Abstracts/
      AbstractContainer.php     ← Abstracts/ ディレクトリは禁止
    Container.php

src/
  Contracts/                    ← トップレベル Contracts/ も禁止
    ContainerInterface.php
    LoggerInterface.php
```

### 7.3 コンポーネント別サブディレクトリ分割規則

単一ディレクトリにファイルが増加した場合のサブディレクトリ分割基準を定める。

| 規則 | 準拠レベル | 内容 |
|------|----------|------|
| 同一ディレクトリに **10ファイルを超える**場合は機能単位でサブディレクトリに分割する | **推奨** | — |
| サブディレクトリへの分割後も名前空間とディレクトリの一致を維持する | **必須** | — |
| 分割基準はファイル数ではなく**責務の区分**を優先する | **推奨** | — |

## 8. 複数ルートマッピング規則（AFE独自）

### 8.1 名前空間プレフィックスの上限（**PSR-4との主要差別化点**）

PSR-4 は `autoload.psr-4` に定義できる名前空間プレフィックスの数を制限しないが、
AFE-STD-101 は**最大3プレフィックスまで**を上限とする。

これは以下の設計目標を達成するためである。

- 名前空間構造の肥大化・分散を防ぎ、コードベースの見通しを保つ
- モノリシックな構造からモジュール分離への移行時の設計複雑度を抑制する
- AFE のモジュラーコンポーネント駆動設計において、各モジュールが `src/` 配下に収まるよう誘導する

| 規則 | 準拠レベル | 内容 |
|------|----------|------|
| `autoload.psr-4` に定義できる名前空間プレフィックスは**最大3つ**とする | **必須** | 4つ以上の定義は禁止 |
| `autoload-dev.psr-4` に定義できるプレフィックスは**最大2つ**とする | **必須** | — |

### 8.2 複数ルートマッピングの許可パターン

以下のパターンのみを許可する。

```json
{
    "autoload": {
        "psr-4": {
            "Adlaire\\Framework\\": "src/",
            "Adlaire\\Shared\\": "shared/",
            "AdlaireGroup\\AdlaireCms\\": "cms-src/"
        }
    },
    "autoload-dev": {
        "psr-4": {
            "Adlaire\\Framework\\Tests\\": "tests/",
            "Adlaire\\Shared\\Tests\\": "shared-tests/"
        }
    }
}
```

## 9. テストコードのオートローディング規則

### 9.1 テストディレクトリ構造

テストコードのディレクトリ構造は、`src/` に対応する `tests/` 内の構造と**完全に一致**させる。

| 規則 | 準拠レベル | 内容 |
|------|----------|------|
| `tests/` 内のディレクトリ構造は `src/` と完全に対応させる | **必須** | — |
| テストクラスの名前空間プレフィックスは `{プロダクション名前空間}Tests\` とする | **必須** | `Adlaire\Framework\` → `Adlaire\Framework\Tests\` |
| テストクラスのファイル名は `{テスト対象クラス名}Test.php` とする | **必須** | AFE-STD-100 Section 11 参照 |

```
対応するプロダクション / テストのマッピング例:

src/Http/Message/Request.php
→ tests/Http/Message/RequestTest.php

src/Container/Container.php
→ tests/Container/ContainerTest.php

src/Logger/Logger.php
→ tests/Logger/LoggerTest.php
```

### 9.2 テストのオートローディング設定

```json
{
    "autoload-dev": {
        "psr-4": {
            "Adlaire\\Framework\\Tests\\": "tests/"
        }
    }
}
```

## 10. 禁止事項

### 10.1 オートローディングに関する禁止事項一覧

| 禁止事項 | 禁止レベル | 代替手段 |
|---------|----------|--------|
| `spl_autoload_register()` による手動オートローダー実装 | **禁止** | Composer `autoload.psr-4` を使用する |
| `require` / `include` によるクラスファイルの手動読み込み | **禁止** | Composer オートローダーで解決する |
| `autoload.classmap` の使用 | **禁止** | `autoload.psr-4` に統一する |
| `Contracts/` / `Interfaces/` / `Abstracts/` / `Base/` ディレクトリの作成 | **禁止** | 実装クラスと同一ディレクトリに配置する |
| `src/` 以外のルートディレクトリ名（プロダクションコード） | **禁止** | `src/` を使用する |
| `tests/` 以外のルートディレクトリ名（テストコード） | **禁止** | `tests/` を使用する |
| スネークケース・ケバブケース・全小文字のディレクトリ名 | **禁止** | StudlyCaps を使用する |
| 4つ以上の `autoload.psr-4` プレフィックス定義 | **禁止** | 最大3プレフィックスに収める |
| 名前空間とディレクトリ構造の不一致 | **禁止** | 完全一致を維持する |

## 11. オートローディング規則 早見表

| 項目 | AFE-STD-101 規則 |
|------|----------------|
| **プロダクションコードルート** | `src/` （固定・変更禁止） |
| **テストコードルート** | `tests/` （固定・変更禁止） |
| **サブディレクトリ命名** | StudlyCaps（名前空間セグメントと完全一致） |
| **名前空間とディレクトリ** | 完全一致（大文字小文字含む） |
| **AFEコアルート名前空間** | `Adlaire\Framework\` → `src/` |
| **オートローダー実装** | Composer のみ（手動実装禁止） |
| **プロダクション環境** | `composer dump-autoload --optimize` |
| **インターフェース配置** | 実装クラスと同一ディレクトリ |
| **抽象クラス配置** | 実装クラスと同一ディレクトリ |
| **Contracts/ 等の専用ディレクトリ** | 禁止 |
| **composer.json プレフィックス数** | autoload 最大3・autoload-dev 最大2 |
| **テスト名前空間** | `{プロダクション名前空間}Tests\` |
| **テストディレクトリ構造** | `src/` と完全対応 |
