## 1. フレームワーク概要

### 1.1 位置づけ

本仕様書は、Adlaire Framework Ecosystem（以下 AFE）のフレームワーク設計・仕様要件を定義するドキュメントである。
上位ドキュメントであるプロジェクト憲章（AFE-CHARTER-001）に基づき、すべての設計・実装・テストはここで定義された仕様に従うものとする。

### 1.2 目的

AFE は Adlaire Group が関与するすべてのプロジェクトおよび内部システムの共通基盤として機能するフレームワークである。
現フェーズでは PHP 8.2 を採用し、将来の開発進捗に応じて Go 言語との統合（ハイブリッド構造）への移行を計画する。

### 1.3 技術スタック（現フェーズ）

| 項目 | 採用技術 |
|------|---------|
| 主要言語 | PHP 8.2 以上 |
| 将来言語 | Go（ハイブリッド移行フェーズにて採用予定） |
| パッケージ管理 | Composer |
| バージョン管理 | Git |
| 実行環境 | PHP-FPM + Nginx または PHP-FPM + Apache |
| **標準仕様** | **AFE-STD（AFE 独自標準仕様）— 外部標準（PSR 等）への依存なし** |

## 2. 設計思想

### 2.1 AFE 三大設計思想

AFE は以下の三大設計思想を核として設計・構築される。
これらは互いに補完関係にあり、一体として機能する。

#### 2.1.1 モジュラーコンポーネント駆動型モデルアーキテクチャ

全機能を独立したコンポーネント単位で構築・管理する。
各コンポーネントは他のコンポーネントへの依存を最小限に抑え、単独での差し替え・拡張・テストが可能な設計とする。

| 原則 | 内容 |
|------|------|
| 単一責任 | 各コンポーネントは単一の責務のみを持つ |
| 疎結合 | コンポーネント間の依存はインターフェース経由に限定する |
| 差し替え可能性 | コンポーネントは実装を入れ替え可能な設計とする |
| 独立テスト性 | 各コンポーネントは他コンポーネントなしでテスト可能とする |

#### 2.1.2 APIインターフェース駆動型モデルアーキテクチャ

すべての機能境界をAPIインターフェースとして定義する。
内部実装の詳細は隠蔽し、インターフェースのみを公開する設計を原則とする。
将来のGo言語との統合においても、このインターフェース設計が言語間連携の基盤となる。

| 原則 | 内容 |
|------|------|
| インターフェース優先 | 実装クラスより先にインターフェースを定義する |
| 実装隠蔽 | 内部実装の詳細を外部に露出しない |
| 言語非依存の境界 | 将来のGoハイブリッド化を見据えたインターフェース設計 |

#### 2.1.3 DIコンテナ・サービスコンテナ駆動型モデルアーキテクチャ

依存関係の解決・コンポーネントの生成・ライフサイクル管理をDIコンテナが一元管理する。
コンポーネント間の依存はDIコンテナを通じて注入され、直接インスタンス化を原則として禁止する。

| 原則 | 内容 |
|------|------|
| 依存の一元管理 | すべての依存解決をDIコンテナが担う |
| コンストラクタインジェクション優先 | 依存の注入はコンストラクタ経由を基本とする |
| サービス登録の明示化 | サービスバインディングは明示的に登録・宣言する |
| スコープ管理 | シングルトン・トランジェント・スコープドの明示的な管理 |

### 2.2 フレームワーク基本哲学

| 哲学 | 内容 |
|------|------|
| **標準準拠** | AFE 独自標準仕様（**AFE-STD**）を唯一の標準体系として全面採用する。外部標準（PHP-FIG PSR 等）への準拠は行わない。すべてのコンポーネント・インターフェースは AFE-STD に従い定義・実装する。AFE-STD は AFE とは独立したプロジェクト [`/AFE-STD/`](../AFE-STD/) として管理される。 |
| **長期安定性** | 後方互換性を重視した設計とする。破壊的変更はメジャーバージョンアップのみに限定する |
| **セキュリティファースト** | セキュリティ対策はオプションではなくフレームワーク層で強制実装する |

## 3. アーキテクチャ設計

### 3.1 全体アーキテクチャ（現フェーズ：PHP 8.2）

```
[ クライアント ]
      ↓
[ Nginx または Apache ]
      ↓
[ PHP-FPM ]
      ↓
[ AFE エントリポイント ]
      ↓
[ ミドルウェアスタック ]
      ↓
[ ルーター ]
      ↓
[ DIコンテナ / サービスコンテナ ]
      ↓
[ アプリケーションレイヤー（コントローラー・サービス・ドメイン） ]
      ↓
[ データレイヤー（ORM・DBレイヤー）]
      ↓
[ データベース / キャッシュ / ストレージ ]
```

### 3.2 レイヤー構成

| レイヤー | 役割 | 主要コンポーネント |
|----------|------|-----------------|
| **エントリポイント層** | リクエストの受付・初期化 | ブートストラップ・設定読み込み |
| **HTTPレイヤー** | リクエスト/レスポンス抽象化 | AFE-STD-200 メッセージ・AFE-STD-202 ミドルウェア |
| **ルーティング層** | URLとハンドラーのマッピング | ルーター・ミドルウェアスタック |
| **DIコンテナ層** | 依存解決・サービス管理 | AFE-STD-300 コンテナ |
| **アプリケーション層** | ビジネスロジック | コントローラー・サービス・ドメインモデル |
| **インフラ層** | 外部システムとの接続 | ORM・キャッシュ・ロガー・メール |

> 📖 ソースコードのディレクトリ構造（レイヤー別）は **AFE-STD-101 Section 7.1** を参照すること。
> 命名規則はすべて **AFE-STD-100** に従う。

### 3.3 コンポーネント依存方向の原則

依存の方向は常に「外側 → 内側」とし、内側のレイヤーは外側のレイヤーに依存しない。
アプリケーション層はインフラ層の具体的実装に直接依存せず、インターフェースを介してのみアクセスする。

## 6. AFE 独自標準仕様（AFE-STD）

### 6.1 概要・方針

AFE は外部標準（PHP-FIG PSR 等）への依存を排し、Adlaire Group が独自に定義・管理する
**AFE-STD（AFE Standard）** を唯一の標準体系として全面採用する。

AFE-STD は **AFE プロジェクトとは独立したプロジェクト** として管理される。

> 📖 AFE-STD の詳細は [`/AFE-STD/`](../AFE-STD/) を参照すること。

| 方針 | 内容 | ステータス |
|------|------|-----------|
| AFE-STD を唯一の標準体系として採用 | 外部標準への依存を排し、AFE 専用の標準仕様を全面適用 | ✅ 確定 |
| PSR への準拠 | ❌ 廃止。いかなる PSR も AFE-STD の代替として使用しない | ❌ 禁止 |
| AFE-STD の独立管理 | AFE-STD は独立プロジェクト `/AFE-STD/` として管理 | ✅ 確定 |
| AFE-STD の段階的策定 | Phase 1〜4 にわたり順次策定・確定 | 🔜 将来予定 |

### 6.2 AFE-STD カテゴリ体系

AFE-STD はカテゴリ別に3桁番号で管理される。

| カテゴリ | 番号帯 | 対象領域 |
|---------|--------|---------|
| **コア標準** | 1xx | 命名規則・オートローディング・コーディングスタイル |
| **HTTPレイヤー標準** | 2xx | HTTPメッセージ・ファクトリー・ミドルウェア・クライアント |
| **コンテナ・DI標準** | 3xx | 依存性注入・サービスコンテナ |
| **ロギング・イベント・時刻標準** | 4xx | ロガー・イベントディスパッチャー・クロック |
| **キャッシュ標準** | 5xx | キャッシュIF・シンプルキャッシュ |
| **アプリケーション標準** | 6xx | ORM・認証認可・バリデーション・CLI・Go連携 |

### 6.5 AFE-STD-100 命名規則 — フレームワーク適用要点

> 📖 完全な仕様は [`/AFE-STD/standards/AFE-STD-100.md`](../AFE-STD/standards/AFE-STD-100.md) を参照すること。

AFE-STD-100 は AFE のすべてのソースコードに適用される命名規則の標準仕様書である。
PSR-1 / PSR-12 と比較して以下の独自規則を定める。

#### 6.5.1 PSR-1 / PSR-12 との主要差別化点

| 項目 | PSR（参考） | AFE-STD-100 |
|------|-----------|------------|
| **インターフェース命名** | 未規定 | `{Name}Interface` サフィックス **必須** |
| **抽象クラス命名** | 未規定 | `Abstract{Name}` プレフィックス **必須** |
| **トレイト命名** | 未規定 | `{Name}Trait` サフィックス **必須** |
| **例外クラス命名** | 未規定 | `{Name}Exception` サフィックス **必須** |
| **列挙型命名（PHP 8.1+）** | 未規定 | `{Name}Enum` サフィックス **必須** ／ ケースは `SCREAMING_SNAKE_CASE` |
| **AFEコンポーネントパターン** | 未規定 | 専用サフィックス 7種 **必須** |
| **`declare(strict_types=1)`** | 未規定 | 全 PHP ファイルで **必須** |
| **パラメータ型宣言** | 未規定 | 全パラメータで **必須** |
| **戻り値型宣言** | 未規定 | 全メソッド・関数で **必須** |
| **`mixed` 型使用** | 未規定 | **禁止**（Union 型・`void`・`never` で代替） |
| **`define()` グローバル定数** | 未規定 | **禁止**（クラス定数・Enum のみ使用可） |
| **テストメソッド命名** | 未規定 | `test{Method}_{Condition}_{Expected}` **必須** |

#### 6.5.2 AFEコンポーネントパターン命名規則（7種）

| コンポーネント | 必須サフィックス | 例 |
|--------------|--------------|----|
| サービスプロバイダー | `ServiceProvider` | `AuthServiceProvider` |
| ミドルウェア | `Middleware` | `CsrfMiddleware` |
| コントローラー | `Controller` | `UserController` |
| リポジトリ | `Repository` | `UserRepository` |
| CLIコマンド | `Command` | `MigrateCommand` |
| イベント | `Event` | `UserCreatedEvent` |
| イベントリスナー | `Listener` | `SendWelcomeMailListener` |

#### 6.5.3 ファイル・名前空間規則要点

```php
<?php
// ✅ AFE-STD-100 準拠の PHP ファイル冒頭
declare(strict_types=1);

namespace Adlaire\Framework\Http\Message;

// インターフェース → {Name}Interface
interface RequestInterface { }

// 抽象クラス → Abstract{Name}
abstract class AbstractMessage { }

// 実装クラス → StudlyCaps（具体的な責務を表す名詞）
class Request extends AbstractMessage implements RequestInterface { }

// 例外 → {Name}Exception
class InvalidRequestException extends AbstractAfeException { }

// Enum → {Name}Enum、ケースは SCREAMING_SNAKE_CASE
enum HttpMethodEnum: string
{
    case GET    = 'GET';
    case POST   = 'POST';
    case DELETE = 'DELETE';
}
```

### 6.6 AFE-STD-101 オートローディング — フレームワーク適用要点

> 📖 完全な仕様は [`/AFE-STD/standards/AFE-STD-101.md`](../AFE-STD/standards/AFE-STD-101.md) を参照すること。

AFE-STD-101 は AFE のソースコードのファイル配置・名前空間マッピング・オートローダー実装方針を定める標準仕様書である。
PSR-4 と比較して以下の独自規則を定める。

#### 6.6.1 PSR-4 との主要差別化点

| 項目 | PSR-4（参考） | AFE-STD-101 |
|------|-------------|------------|
| **ルートディレクトリ名** | 任意 | `src/` 固定・変更禁止 |
| **テストディレクトリ名** | 任意 | `tests/` 固定・変更禁止 |
| **サブディレクトリ命名** | 任意 | StudlyCaps 必須（AFE-STD-100 連動） |
| **レイヤー別ディレクトリ構造** | 未規定 | 7レイヤー固定構造を必須定義 |
| **インターフェース配置** | 未規定（`Contracts/` 等が慣習） | 実装クラスと**同一ディレクトリ**必須 |
| **抽象クラス配置** | 未規定（`Abstracts/` 等が慣習） | 実装クラスと**同一ディレクトリ**必須 |
| **`Contracts/` / `Interfaces/` / `Abstracts/` / `Base/` ディレクトリ** | 未規定（広く使われる慣習） | **明示的に禁止** |
| **オートローダー実装手段** | PSR-4 準拠なら任意 | **Composer のみ**（手動実装禁止） |
| **`autoload.psr-4` プレフィックス数** | 制限なし | 最大 3（`autoload-dev` は最大 2） |
| **プロダクション環境コマンド** | 未規定 | `composer dump-autoload --optimize` 必須 |

#### 6.6.2 ソースコードディレクトリ構造（必須）

```
project-root/
├── src/                       ← プロダクションコード（固定・変更禁止）
│   ├── Http/                  ← HTTPレイヤー（AFE-STD-200〜202）
│   │   ├── Message/           ← リクエスト・レスポンス
│   │   ├── Middleware/        ← ミドルウェア群
│   │   └── Factory/           ← HTTPオブジェクトファクトリー
│   ├── Container/             ← DIコンテナ（AFE-STD-300）
│   ├── Logger/                ← ロガー（AFE-STD-400）
│   ├── Event/                 ← イベントシステム（AFE-STD-401）
│   ├── Cache/                 ← キャッシュ（AFE-STD-500〜501）
│   ├── Database/              ← ORM・DB（AFE-STD-600）
│   │   └── Orm/
│   ├── Auth/                  ← 認証・認可（AFE-STD-601）
│   │   └── Rbac/
│   ├── Validation/            ← バリデーション（AFE-STD-602）
│   ├── Console/               ← CLIツール（AFE-STD-603）
│   ├── Exception/             ← フレームワーク共通例外
│   └── Support/               ← ユーティリティ・ヘルパー
├── tests/                     ← テストコード（src/ と完全対応）
└── composer.json
```

#### 6.6.3 インターフェース・抽象クラスの配置規則

PSR-4 慣習（`Contracts/`・`Abstracts/` 等）は AFE では**全面禁止**。
インターフェース・抽象クラスは実装クラスと**同一ディレクトリ**に配置する。

```
✅ AFE-STD-101 準拠
src/Container/
  ContainerInterface.php      ← インターフェース（同一ディレクトリ）
  AbstractContainer.php       ← 抽象クラス（同一ディレクトリ）
  Container.php               ← 実装クラス

❌ 禁止（PSR-4 慣習）
src/Container/
  Contracts/ContainerInterface.php    ← Contracts/ ディレクトリは禁止
  Abstracts/AbstractContainer.php     ← Abstracts/ ディレクトリは禁止
```

#### 6.6.4 composer.json オートローディング設定（標準形）

```json
{
    "autoload": {
        "psr-4": {
            "Adlaire\\Framework\\": "src/"
        }
    },
    "autoload-dev": {
        "psr-4": {
            "Adlaire\\Framework\\Tests\\": "tests/"
        }
    }
}
```

> ⚠️ `autoload.psr-4` キー名は Composer の設定インターフェース上の表記であり、
> PSR-4 への準拠を意味しない。オートローディングの実質的な規則は AFE-STD-101 が定義する。

## 7. セキュリティ仕様

セキュリティ対策はオプションではなく、フレームワーク層で強制実装する。

| 脅威 | 対策 | 実装層 |
|------|------|-------|
| **SQLインジェクション** | プリペアドステートメントの強制・クエリビルダーによるエスケープ | ORM・DBレイヤー |
| **XSS（クロスサイトスクリプティング）** | 出力エスケープのデフォルト有効化 | テンプレート・レスポンス層 |
| **CSRF（クロスサイトリクエストフォージェリ）** | CSRFトークンの自動生成・検証 | ミドルウェア層 |
| **セッションハイジャック** | セッションIDの再生成・セキュアフラグ | 認証コンポーネント |
| **パスワード管理** | bcrypt または Argon2 によるハッシュ化の強制 | 認証コンポーネント |
| **ディレクトリトラバーサル** | パス正規化・ホワイトリスト検証 | ファイル処理層 |
| **機密情報の露出** | エラーメッセージの制御・スタックトレースの非公開化 | エラーハンドラー |

準拠基準: **OWASP Top 10**

## 9. 拡張・プラグイン仕様

### 9.1 拡張方針

AFE はモジュラーコンポーネント設計により、コア機能を変更することなく機能を追加・差し替え可能な拡張機構を提供する。

| 拡張種別 | 内容 |
|----------|------|
| **サービスプロバイダー** | DIコンテナへのサービス登録・初期化処理の拡張 |
| **ミドルウェア** | HTTPパイプラインへの処理の追加 |
| **イベントリスナー** | フレームワーク・アプリケーションイベントへのフック |
| **コマンド** | CLIコマンドの追加 |
| **コンポーネント差し替え** | デフォルト実装を独自実装に置換 |

### 9.2 独自コンポーネント登録規約

拡張・プラグインの登録は AFE-STD に従った形式で行い、フレームワークコアへの直接依存を禁止する。
すべての拡張はインターフェースを実装する形で提供しなければならない。

## 11. 将来ロードマップ（PHPとGoのハイブリッド構造）

### 11.1 ハイブリッド化の方針

当初の検討段階において、以下の3つのアーキテクチャパターンを想定している。
いずれのパターンを採用するかは、今後の開発進捗・技術検証（POC）・チームスキル・運用要件に基づいて決定する。

### 11.2 検討中アーキテクチャパターン

#### パターン A：ゲートウェイ分離型

Goが高速HTTPゲートウェイを担当し、PHPがビジネスロジックを処理する構造。

| レイヤー | 担当言語 | 役割 |
|----------|---------|------|
| HTTPゲートウェイ | Go | リクエストルーティング・認証検証・レートリミット・ログ収集 |
| ビジネスロジック | PHP | ドメインロジック・ORM・業務処理 |

- **メリット**: PHP資産を最大保持しつつエントリポイントを高速化
- **適用場面**: 高トラフィック対応が必要な場合

#### パターン B：サービス分離型（gRPC通信）

処理特性に応じてPHPとGoを役割分担し、gRPCで通信する構造。

| サービス | 担当言語 | 役割 |
|----------|---------|------|
| 業務サービス | PHP | ドメインロジック・CRUD・認証・権限管理 |
| 高性能サービス | Go | 高並行処理・リアルタイム・バッチ・外部API集約 |

- **メリット**: 各言語の得意領域を活かした最適分業。段階的なGo移行が可能
- **適用場面**: マイクロサービス化が必要な場合

#### パターン C：FrankenPHPワーカー統合型

FrankenPHP（Go製PHPサーバー）によりPHPとGoを単一実行環境で統合する構造。

| レイヤー | 内容 |
|----------|------|
| HTTPサーバー | FrankenPHP（Go + Caddy） |
| PHP実行 | AFEコア（PHP 8.2）をワーカーモードで常駐 |
| 将来拡張 | Go拡張モジュールとの統合 |

- **メリット**: 単一バイナリ・HTTP/3対応・TLS自動管理。最も自然なハイブリッド移行パス
- **適用場面**: 新規プロジェクト・インフラ簡素化を重視する場合

### 11.3 フェーズ計画

| フェーズ | 時期 | 実行環境 | 言語構成 |
|----------|------|---------|---------|
| 現フェーズ | 〜 2026 Q3 | PHP-FPM + Nginx または Apache | PHP 8.2 |
| 検討・POCフェーズ | 2026 Q4 | パターン A/B/C を技術検証 | PHP 8.2 + Go（検証） |
| 移行フェーズ | 2027 Q1〜 | 採用パターンで本番移行 | PHP 8.2 + Go（本番） |
