## 1. フレームワーク概要

### 1.1 位置づけ

本仕様書は、Adlaire Framework Ecosystem（以下 AFE）のフレームワーク設計・仕様要件を定義するドキュメントである。
上位ドキュメントであるプロジェクト憲章（AFE-CHARTER-001）に基づき、すべての設計・実装・テストはここで定義された仕様に従うものとする。

### 1.2 目的

AFE は Adlaire Group が関与するすべてのプロジェクトおよび内部システムの共通基盤として機能するフレームワークである。
現フェーズでは PHP 8.2 を採用し、将来の開発進捗に応じて Go 言語との統合（ハイブリッド構造）への移行を計画する。

### 1.3 技術スタック（現フェーズ）

| 項目 | 採用技術 |
|------|---------|
| 主要言語 | PHP 8.2 以上 |
| 将来言語 | Go（ハイブリッド移行フェーズにて採用予定） |
| パッケージ管理 | Composer |
| バージョン管理 | Git |
| 実行環境 | PHP-FPM + Nginx または PHP-FPM + Apache |

## 2. 設計思想

### 2.1 AFE 三大設計思想

AFE は以下の三大設計思想を核として設計・構築される。
これらは互いに補完関係にあり、一体として機能する。

#### 2.1.1 モジュラーコンポーネント駆動型モデルアーキテクチャ

全機能を独立したコンポーネント単位で構築・管理する。
各コンポーネントは他のコンポーネントへの依存を最小限に抑え、単独での差し替え・拡張・テストが可能な設計とする。

| 原則 | 内容 |
|------|------|
| 単一責任 | 各コンポーネントは単一の責務のみを持つ |
| 疎結合 | コンポーネント間の依存はインターフェース経由に限定する |
| 差し替え可能性 | コンポーネントは実装を入れ替え可能な設計とする |
| 独立テスト性 | 各コンポーネントは他コンポーネントなしでテスト可能とする |

#### 2.1.2 APIインターフェース駆動型モデルアーキテクチャ

すべての機能境界をAPIインターフェースとして定義する。
内部実装の詳細は隠蔽し、インターフェースのみを公開する設計を原則とする。
将来のGo言語との統合においても、このインターフェース設計が言語間連携の基盤となる。

| 原則 | 内容 |
|------|------|
| インターフェース優先 | 実装クラスより先にインターフェースを定義する |
| 実装隠蔽 | 内部実装の詳細を外部に露出しない |
| 言語非依存の境界 | 将来のGoハイブリッド化を見据えたインターフェース設計 |

#### 2.1.3 DIコンテナ・サービスコンテナ駆動型モデルアーキテクチャ

依存関係の解決・コンポーネントの生成・ライフサイクル管理をDIコンテナが一元管理する。
コンポーネント間の依存はDIコンテナを通じて注入され、直接インスタンス化を原則として禁止する。

| 原則 | 内容 |
|------|------|
| 依存の一元管理 | すべての依存解決をDIコンテナが担う |
| コンストラクタインジェクション優先 | 依存の注入はコンストラクタ経由を基本とする |
| サービス登録の明示化 | サービスバインディングは明示的に登録・宣言する |
| スコープ管理 | シングルトン・トランジェント・スコープドの明示的な管理 |

### 2.2 フレームワーク基本哲学

| 哲学 | 内容 |
|------|------|
| **標準準拠** | PHP-FIG の PSR 標準を全面採用する。PSR 標準が存在しない領域については AFE 独自標準準拠仕様を策定し適用する |
| **長期安定性** | 後方互換性を重視した設計とする。破壊的変更はメジャーバージョンアップのみに限定する |
| **セキュリティファースト** | セキュリティ対策はオプションではなくフレームワーク層で強制実装する |

## 3. アーキテクチャ設計

### 3.1 全体アーキテクチャ（現フェーズ：PHP 8.2）

```
[ クライアント ]
      ↓
[ Nginx または Apache ]
      ↓
[ PHP-FPM ]
      ↓
[ AFE エントリポイント ]
      ↓
[ ミドルウェアスタック ]
      ↓
[ ルーター ]
      ↓
[ DIコンテナ / サービスコンテナ ]
      ↓
[ アプリケーションレイヤー（コントローラー・サービス・ドメイン） ]
      ↓
[ データレイヤー（ORM・DBレイヤー）]
      ↓
[ データベース / キャッシュ / ストレージ ]
```

### 3.2 レイヤー構成

| レイヤー | 役割 | 主要コンポーネント |
|----------|------|-----------------|
| **エントリポイント層** | リクエストの受付・初期化 | ブートストラップ・設定読み込み |
| **HTTPレイヤー** | リクエスト/レスポンス抽象化 | PSR-7 メッセージ・PSR-15 ミドルウェア |
| **ルーティング層** | URLとハンドラーのマッピング | ルーター・ミドルウェアスタック |
| **DIコンテナ層** | 依存解決・サービス管理 | PSR-11 コンテナ |
| **アプリケーション層** | ビジネスロジック | コントローラー・サービス・ドメインモデル |
| **インフラ層** | 外部システムとの接続 | ORM・キャッシュ・ロガー・メール |

### 3.3 コンポーネント依存方向の原則

依存の方向は常に「外側 → 内側」とし、内側のレイヤーは外側のレイヤーに依存しない。
アプリケーション層はインフラ層の具体的実装に直接依存せず、インターフェースを介してのみアクセスする。

## 7. セキュリティ仕様

セキュリティ対策はオプションではなく、フレームワーク層で強制実装する。

| 脅威 | 対策 | 実装層 |
|------|------|-------|
| **SQLインジェクション** | プリペアドステートメントの強制・クエリビルダーによるエスケープ | ORM・DBレイヤー |
| **XSS（クロスサイトスクリプティング）** | 出力エスケープのデフォルト有効化 | テンプレート・レスポンス層 |
| **CSRF（クロスサイトリクエストフォージェリ）** | CSRFトークンの自動生成・検証 | ミドルウェア層 |
| **セッションハイジャック** | セッションIDの再生成・セキュアフラグ | 認証コンポーネント |
| **パスワード管理** | bcrypt または Argon2 によるハッシュ化の強制 | 認証コンポーネント |
| **ディレクトリトラバーサル** | パス正規化・ホワイトリスト検証 | ファイル処理層 |
| **機密情報の露出** | エラーメッセージの制御・スタックトレースの非公開化 | エラーハンドラー |

準拠基準: **OWASP Top 10**

## 9. 拡張・プラグイン仕様

### 9.1 拡張方針

AFE はモジュラーコンポーネント設計により、コア機能を変更することなく機能を追加・差し替え可能な拡張機構を提供する。

| 拡張種別 | 内容 |
|----------|------|
| **サービスプロバイダー** | DIコンテナへのサービス登録・初期化処理の拡張 |
| **ミドルウェア** | HTTPパイプラインへの処理の追加 |
| **イベントリスナー** | フレームワーク・アプリケーションイベントへのフック |
| **コマンド** | CLIコマンドの追加 |
| **コンポーネント差し替え** | デフォルト実装を独自実装に置換 |

### 9.2 独自コンポーネント登録規約

拡張・プラグインの登録は AFE-STD に従った形式で行い、フレームワークコアへの直接依存を禁止する。
すべての拡張はインターフェースを実装する形で提供しなければならない。

## 11. 将来ロードマップ（PHPとGoのハイブリッド構造）

### 11.1 ハイブリッド化の方針

当初の検討段階において、以下の3つのアーキテクチャパターンを想定している。
いずれのパターンを採用するかは、今後の開発進捗・技術検証（POC）・チームスキル・運用要件に基づいて決定する。

### 11.2 検討中アーキテクチャパターン

#### パターン A：ゲートウェイ分離型

Goが高速HTTPゲートウェイを担当し、PHPがビジネスロジックを処理する構造。

| レイヤー | 担当言語 | 役割 |
|----------|---------|------|
| HTTPゲートウェイ | Go | リクエストルーティング・認証検証・レートリミット・ログ収集 |
| ビジネスロジック | PHP | ドメインロジック・ORM・業務処理 |

- **メリット**: PHP資産を最大保持しつつエントリポイントを高速化
- **適用場面**: 高トラフィック対応が必要な場合

#### パターン B：サービス分離型（gRPC通信）

処理特性に応じてPHPとGoを役割分担し、gRPCで通信する構造。

| サービス | 担当言語 | 役割 |
|----------|---------|------|
| 業務サービス | PHP | ドメインロジック・CRUD・認証・権限管理 |
| 高性能サービス | Go | 高並行処理・リアルタイム・バッチ・外部API集約 |

- **メリット**: 各言語の得意領域を活かした最適分業。段階的なGo移行が可能
- **適用場面**: マイクロサービス化が必要な場合

#### パターン C：FrankenPHPワーカー統合型

FrankenPHP（Go製PHPサーバー）によりPHPとGoを単一実行環境で統合する構造。

| レイヤー | 内容 |
|----------|------|
| HTTPサーバー | FrankenPHP（Go + Caddy） |
| PHP実行 | AFEコア（PHP 8.2）をワーカーモードで常駐 |
| 将来拡張 | Go拡張モジュールとの統合 |

- **メリット**: 単一バイナリ・HTTP/3対応・TLS自動管理。最も自然なハイブリッド移行パス
- **適用場面**: 新規プロジェクト・インフラ簡素化を重視する場合
