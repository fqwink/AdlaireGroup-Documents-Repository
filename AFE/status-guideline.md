## 1. 目的

Adlaire Group DX 事業セグメントが管理するすべてのドキュメント（設計書・憲章・仕様書等）において、未決定事項・進行中事項のステータス表記が人によってバラバラになることを防ぐ。

本ガイドラインは、**ステータス記号の意味・使い分け基準・遷移ルール**を明文化し、ドキュメントの読み手が一目で「この項目が今どの状態にあるか」を正確に把握できるようにすることを目的とする。

### 適用範囲

- `PROJECT_CHARTER.md`（プロジェクト憲章）
- `FRAMEWORK_SPECIFICATION.md`（フレームワーク仕様書）
- `VERSIONING.md`（バージョニング規約）
- `design-vX.X.md`（設計書）
- `label-map.md`（用語マップ）
- その他 Adlaire Group DX 事業セグメントが作成・管理するすべての Markdown ドキュメント

## 2. ステータス一覧

| 記号 | ステータス名 | 一言定義 |
|------|------------|---------|
| ✅ | **確定** | 確定済み。変更しない |
| ⚠️ | **暫定** | 仮方針・仮運用中。変更可能性あり |
| 🔧 | **策定中** | 方針を検討中・議論している最中 |
| 🔜 | **将来予定** | 特定フェーズ・時期に着手予定。方針は固まっている |
| ❓ | **未定** | 着手していない。方針が存在しない |
| ❌ | **禁止** | 採用・実装・変更を禁ずる |
| — | **方針なし** | 現時点で定義対象外。決定すべき事項ではない |

## 3. 各ステータスの定義と使い分け基準

---

### ✅ 確定

#### 定義

意思決定が完了しており、**原則として変更しない**項目。

#### 使用条件（すべてを満たすこと）

- [ ] 関係者間で合意・承認が得られている
- [ ] ドキュメントに明文化されている
- [ ] 実装・運用の根拠として参照できる状態にある

#### 典型例

```
PHP 8.2 以上を必須要件とする       → ✅ 確定
エディタに Editor.js を採用する    → ✅ 確定
サードパーティ製を禁止する → ✅ 確定（❌ 禁止事項と併用禁止事項）
```

#### 注意

- 確定後に変更が生じた場合は、変更履歴に理由を記録した上で再確定（再度 ✅）とする
- 「なんとなく決まっている気がする」は ✅ ではなく ⚠️ または 🔧 を使う

---

### ⚠️ 暫定

#### 定義

**現時点では仮の方針で運用している**が、将来的に変更・正式確定される可能性がある項目。「とりあえず動かすための仮決め」。

#### 使用条件（いずれかを満たすこと）

- 正式な合意はないが、開発を進めるために仮の方針が必要な場合
- 方針は存在するが、外部要因（販売戦略・クライアント要件等）によって変わりうる場合
- 仮の値・仮の設計で実装が先行している場合

#### ✅ 確定 との違い

| | ✅ 確定 | ⚠️ 暫定 |
|-|--------|--------|
| 合意の有無 | 正式に合意済み | 仮合意・未合意 |
| 変更可能性 | 原則なし | あり |
| 運用状態 | そのまま運用 | 仮運用中 |

#### 典型例

```
プロジェクト名称: Adlaire CMS（正式名称確定前の仮称）     → ⚠️ 暫定
```

#### 注意

- 暫定項目には必ず「何が確定したら ✅ になるか」を備考欄に記載することを推奨する
- 暫定のまま長期放置することを避ける。定期的に見直し、確定か方針変更かを判断する

---

### 🔧 策定中

#### 定義

**能動的に検討・議論・設計が進行中**の項目。担当者が動いており、近い将来に結論が出る見込みがある。

#### 使用条件（すべてを満たすこと）

- [ ] 検討主体が明確に存在する
- [ ] 何らかの検討・議論・調査が実際に進行している
- [ ] 着地点（✅ 確定 または ⚠️ 暫定）に向けて動いている

#### ⚠️ 暫定 との違い

| | ⚠️ 暫定 | 🔧 策定中 |
|-|--------|---------|
| 現在の状態 | 仮の方針で運用中 | 方針を検討中（まだ運用できない） |
| アクション | 受動的（変更待ち） | 能動的（検討が進行中） |
| 結論の見込み | 不定 | 近い将来に結論が出る見込みあり |

#### 典型例

```
Adlaire Framework Ecosystem の詳細設計: 議論進行中      → 🔧 策定中
テーマ API / テンプレート関数仕様: Sprint 4 で確定予定  → 🔧 策定中
```

#### 注意

- 検討が止まって放置されている場合は ❓ 未定 に戻す
- 「誰かが考えているはずだが自分は知らない」は 🔧 策定中 ではなく ❓ 未定 を使う

---

### 🔜 将来予定

#### 定義

**実施・実装することが決定しており**、特定のフェーズ・時期に着手予定の項目。「やる」ことは確定しているが「今ではない」もの。

#### 使用条件（すべてを満たすこと）

- [ ] 「実施する」という意思決定が完了している
- [ ] 着手予定のフェーズ・時期が明示できる（「Phase 2」「Sprint 4」等）
- [ ] 現在は着手していない

#### ❓ 未定 との違い

| | 🔜 将来予定 | ❓ 未定 |
|-|-----------|--------|
| 「やる」決定 | 済み | なし |
| 時期の見通し | 明示できる | 不明 |
| 優先度 | ロードマップ上に存在 | ロードマップ未掲載 |

#### 典型例

```
セカンドパーティ対応: Phase 4 以降に開放予定               → 🔜 将来予定（Phase 4）
SSG 機能: Phase 4 で実装予定（PHP 製）                     → 🔜 将来予定（Phase 4）
```

#### 注意

- フェーズ・時期を必ず併記する（例: `🔜 将来予定（Phase 2）`）
- 時期が不明な場合は 🔜 ではなく ❓ 未定 を使う
- ロードマップが変更された際は、フェーズ番号を更新する

---

### ❓ 未定

#### 定義

**決定すべき事項が未着手**の項目。将来的に意思決定が必要だが、まだ誰も着手していない、または意図的に後回しにしている状態。

> **`—` 方針なし との違い**: ❓ 未定 は「いずれ決定が必要な事項」。`—` 方針なし は「現時点で定義対象外であり、決定を求めない事項」。

#### 使用条件（いずれかを満たすこと）

- 将来的に決定が必要だが、まだ何も決まっていない（検討・方針・時期すべて不明）
- 意図的に判断を保留している
- 外部要因が確定するまで動けない

#### 他のステータスとの比較

| ステータス | 方針 | 担当 | 時期 |
|-----------|------|------|------|
| ✅ 確定 | あり（確定） | — | — |
| ⚠️ 暫定 | あり（仮） | — | 不定 |
| 🔧 策定中 | 検討中 | あり | 近日 |
| 🔜 将来予定 | あり（確定） | あり | 明示可 |
| **❓ 未定** | **なし** | **不在** | **不明** |
| **— 方針なし** | **対象外** | **—** | **—** |

#### 典型例

```
販売モデル（有償 / 無償 / SaaS）: 将来決定が必要だが未着手  → ❓ 未定
ライセンス形態（OSS / 商用 / デュアル）: 未着手             → ❓ 未定
正式プロジェクト名称: 現時点で検討予定なし                   → ❓ 未定（⚠️ 暫定 と併用可）
MySQL / PostgreSQL: 採用方針なし　　　　　                 → —（方針なし）
```

#### 注意

- ❓ 未定 のまま長期放置される項目は、定期レビュー時に「❓ 未定のまま継続する」か「🔧 策定中 に昇格する」かを判断する
- 「誰かが決めてくれるはず」という状態は ❓ 未定 で管理する

---

### ❌ 禁止

#### 定義

**採用・実装・変更を明示的に禁じる**項目。ルールとして文書化されており、例外は原則認めない。

#### 使用条件（すべてを満たすこと）

- [ ] 禁止の意思決定が正式に合意されている（✅ 確定 との組み合わせが基本）
- [ ] 禁止対象が具体的に明示されている
- [ ] 理由または根拠がドキュメントに記載されている

#### 典型例

```
サードパーティ製拡張の採用    → ❌ 禁止
外部フレームワーク（例外を除く）の使用    → ❌ 禁止
ビルド/リビジョン番号のリセット          → ❌ 禁止
```

#### 注意

- ❌ 禁止 は必ず理由・根拠へのリンクまたは注記を添える
- 例外が存在する場合は「例外」として明示する（例: 例外 A / B / C）

---

### — 方針なし

#### 定義

**現時点で定義対象外**の項目。「決定すべき事項」ではなく、プロジェクトの対象範囲に含まれていない、または方針を定める必要がないと判断された事項。

> **❓ 未定 との違い**: `—` 方針なし は「決定を求めない」。❓ 未定 は「いずれ決定が必要」。

#### 使用条件（いずれかを満たすこと）

- プロジェクトの現在の対象範囲に含まれていない
- 方針を定義すること自体が不要と判断されている
- 採用も禁止もしておらず、単に対象外である

#### 典型例

```
MySQL / PostgreSQL / NoSQL: 採用方針なし → —（方針なし）
```

#### 記述方法

テーブルのステータス列に `—` を記載する。

```markdown
| その他 DB（MySQL / PostgreSQL 等） | — | — |
```

#### 注意

- `—` は「禁止ではない」。必要になれば方針を定義できる状態。
- `—` を長期間維持する場合は、定期レビューで「このままスコープ外でよいか」を確認する。

## 4. ステータス遷移フロー

### 標準遷移パターン

```
新しい項目の発生
       │
   ┌───┴──────────────────────────┐
   │                              │
   │ 定義不要     　　　　　　　    │ 将来的に決定が必要
   ▼                              ▼
  — 方針なし                  ❓ 未定
                              （決定すべき事項・未着手）
       │
       │ 担当者が決まり、検討開始
       ▼
   🔧 策定中
   （能動的に検討進行中）
       │
   ┌───┴────────────────────┐
   │                        │
   │ 結論が出た             │ 「やる、でも今じゃない」
   ▼                        ▼
 ✅ 確定              🔜 将来予定
                      （フェーズ・時期を明示）
                            │
                            │ 着手フェーズが到来
                            ▼
                        🔧 策定中
                            │
                            │ 結論が出た
                            ▼
                          ✅ 確定
```

### 暫定を含む遷移パターン

```
開発を進めるために仮の方針が必要
       │
       ▼
   ⚠️ 暫定
   （仮の方針で運用中）
       │
   ┌───┴─────────────────────┐
   │                         │
   │ 正式合意が得られた       │ 方針が根本から変わる
   ▼                         ▼
 ✅ 確定                  🔧 策定中
                              │
                              ▼
                           ✅ 確定
```

### 禁止への遷移

```
✅ 確定（採用しないと決定）
       │
       ▼
   ❌ 禁止
（理由・根拠を必ず併記）
```

### 逆行（降格）が発生しうるケース

| 現在 | 降格先 | 発生条件 |
|------|--------|---------|
| 🔧 策定中 | ❓ 未定 | 検討が中断・凍結された |
| ⚠️ 暫定 | 🔧 策定中 | 暫定方針を見直す必要が生じた |
| ✅ 確定 | 🔧 策定中 | 前提条件の変化により再検討が必要になった |

> ⚠️ 確定から降格する場合は、**変更履歴に理由を必ず記録する**。

## 5. 記述ルール

### 5-1. 記述形式

#### テーブル内での使用

```markdown
| 項目 | 内容 | ステータス |
|------|------|-----------|
| PHP バージョン | PHP 8.2 以上 | ✅ 確定 |
| 管理画面設置方式 | レンタルサーバー上（暫定） | ⚠️ 暫定 |
| 画像アップロード | サーバー保存・GitHub 連携 | 🔜 将来予定（Phase 2） |
| 販売モデル | — | ❓ 未定 |
```

#### インライン（文章中）での使用

```markdown
GitHub 連携方式は 🔧 策定中 であり、内製実装とライブラリ利用を比較検討している。
SSG 機能は 🔜 将来予定（Phase 4）として計画されている。
```

#### コードブロック内（jsonc / コメント）での使用

```jsonc
"ssg": {
    "_status": "planned",   // 🔜 将来予定（Phase 4）
    "enabled": false
},
"staging": {
    "_status": "planned"    // 🔜 将来予定（Phase 2）
}
```

### 5-2. 補足情報の付記ルール

| ステータス | 必須補足 | 推奨補足 |
|-----------|---------|---------|
| ✅ 確定 | — | 確定日・合意者 |
| ⚠️ 暫定 | 「何が確定したら✅になるか」 | 期限・見直し時期 |
| 🔧 策定中 | 担当者または検討主体 | 結論予定時期 |
| 🔜 将来予定 | **フェーズ・時期（必須）** | 担当者・前提条件 |
| ❓ 未定 | — | 「いつ着手するかの目安」 |
| ❌ 禁止 | **理由・根拠（必須）** | 参照ドキュメント |
| — 方針なし | — | 「いつ対象範囲に入るか」の目安 |

### 5-3. ステータス列の配置

- テーブルでは原則として**最右列**にステータス列を設ける
- 列名は「ステータス」で統一する
- ステータスが自明な場合（全行が ✅ の場合等）は列を省略してもよい

### 5-4. 複合ステータスの使い方

1つの項目に複数のステータスが該当する場合は、主ステータスを先に記載し、補足を括弧内に記す。

```markdown
| 正式プロジェクト名称 | Adlaire CMS | ⚠️ 暫定（正式名称は❓ 未定） |
| サードパーティ拡張  | 禁止        | ❌ 禁止（✅ 確定済み）      |
```

## 6. 判定フローチャート

新しい項目・変更をドキュメントに記載する際、以下のフローで適切なステータスを選ぶ。

```
この項目は正式に合意・決定されているか？
├─ YES → ✅ 確定
│         └─ ただし「採用しない」と決定した場合 → ❌ 禁止
│
└─ NO
    │
    ├─ 仮の方針で運用が始まっているか？
    │   └─ YES → ⚠️ 暫定
    │
    ├─ 担当者がいて、能動的に検討中か？
    │   └─ YES → 🔧 策定中
    │
    ├─ 「やる」ことは決まっているが、今は着手しないか？
    │   └─ YES → 🔜 将来予定（フェーズ・時期を必ず明記）
    │
    ├─ 上記のどれにも当てはまらない（将来的に決定が必要）
    │   └─ ❓ 未定
    │
    └─ そもそも方針を定める必要がない
        └─ —（方針なし）
```

## 7. 実例集

### Adlaire CMS プロジェクトにおける適用例

| 項目 | 正しいステータス | 理由 |

| PHP 8.2 必須要件 | ✅ 確定 | 正式決定済み・設計書に明文化 |

| GitHub 連携方式（内製 vs ライブラリ） | 🔧 策定中 | Phase 0 残タスクとして検討進行中 |

| テーマ API / テンプレート関数仕様 | 🔧 策定中 | Sprint 4 で確定予定 |
| 画像アップロード（サーバー保存） | 🔜 将来予定（Phase 2） | 実施決定済み。Phase 2 で着手 |
| SSG 機能 | 🔜 将来予定（Phase 4） | PHP 製で実装することは決定。Phase 4 |
| セカンドパーティ対応 | 🔜 将来予定（Phase 4 以降） | 認定プロセス設計完了後に開放 |
| 販売モデル（有償 / 無償 / SaaS） | ❓ 未定 | 方針なし・担当者なし・時期不明 |
| ライセンス形態 | ❓ 未定 | 同上 |
| WordPress 移行ツール公開範囲 | ❓ 未定 | Phase 3 で検討予定だが方針なし |
| ビルド/リビジョン番号のリセット | ❌ 禁止 | VERSIONING.md に明文化された禁止事項 |
| その他 DB（MySQL / PostgreSQL 等） | — 方針なし | 現時点で。採用も禁止もしない |

### 誤用例 → 正しい例

```
❌ 誤: 管理画面設置方式  →  未定
✅ 正: 管理画面設置方式  →  ⚠️ 暫定
   理由: 仮の方針（レンタルサーバー）で運用中のため「暫定」が正しい。
         方針がゼロの「未定」とは異なる。

❌ 誤: セカンドパーティ対応  →  策定中
✅ 正: セカンドパーティ対応  →  🔜 将来予定（Phase 4 以降）
   理由: 「やる」ことは決定済みだが今は着手しない。
         能動的に検討が動いているわけではないため「策定中」ではない。

❌ 誤: GitHub 連携方式  →  将来予定
✅ 正: GitHub 連携方式  →  🔧 策定中
   理由: Phase 0 の残タスクとして今まさに検討中。
         時期を先送りにしているわけではないため「将来予定」ではない。
```

## 8. よくある誤用と正しい使い方

### Q1. 「暫定」と「未定」はどう違うの？

> **A.** 「暫定（⚠️）」は**仮の方針が存在して運用中**。「未定（❓）」は**方針ゼロ**。
>
> 例: 管理画面をとりあえずレンタルサーバーに置いている → ⚠️ 暫定
>     販売モデルはまだ何も考えていない → ❓ 未定

### Q2. 「策定中」と「将来予定」はどう違うの？

> **A.** 「策定中（🔧）」は**今まさに検討が動いている**。「将来予定（🔜）」は**やることは決まっているが今は着手しない**。
>
> 例: GitHub 連携方式を今週中に決める予定で比較中 → 🔧 策定中
>     Phase 4 で SSG を実装することは決まっている → 🔜 将来予定（Phase 4）

### Q3. 「確定」の項目が変更になったらどうする？

> **A.** 変更履歴に理由を記録したうえで、状況に応じて 🔧 策定中 → 再度 ✅ 確定 とする。
> 一度 ✅ になった項目を黙って書き換えるのは禁止。

### Q4. 「禁止」に理由を書かなくていいケースはある？

> **A.** 原則として理由を書く。ただし、参照先ドキュメントが明確な場合（「VERSIONING.md 参照」等）は参照リンクで代替可能。

### Q5. ステータスが「未定」の項目は設計書に書いてよいの？

> **A.** 書いてよい。むしろ「未定であることを明示する」ために書く。ドキュメントに載せることで「誰も決めていない」ことが可視化され、優先度の判断材料になる。

### Q6. 「未定（❓）」と「方針なし（—）」はどう違うの？

> **A.** 「未定（❓）」は**将来的に決定が必要な事項**。「方針なし（—）」は**現時点で定義対象外であり、決定を求めない事項**。
>
> 例: 販売モデルはいずれ決める必要がある → ❓ 未定
>     MySQL / PostgreSQL は採用も禁止もしていない・方針を定める必要がない → — 方針なし
